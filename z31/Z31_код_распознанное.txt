 Z31 — РАСПОЗНАВАНИЕ НЕИЗВЕСТНЫХ ДАННЫХ ПО КОДУ ПРОШИВКИ
 Прошивка: 88_Z31turbo32k.bin (Nissan 300ZX Z31, VG30ET, 1988).
 Метод: дизассемблировали код процессора (Hitachi HD6303) и смотрели, КТО и
        КАК читает каждый кусок данных — отсюда его смысл.

 КАК ЧИТАТЬ ЭТОТ ФАЙЛ
 - Здесь перечислено ВСЁ, что НЕ было описано в готовом XDF (то, чего мы не
   знали), и что мы выяснили про эти данные из кода.
 - У каждого пункта: АДРЕС, ТИП (карта/ось/кривая/скаляр/опции…), ЧТО ЭТО,
   и СТАТУС: [ДОКАЗАНО] = подтверждено кодом, [ГИПОТЕЗА] = по косвенным
   признакам, точно не подтверждено.
 - Адрес в файле и адрес в процессоре связаны просто: АДРЕС_В_CPU = адрес_в_файле + 0x4000.
   (например, таблица в файле по 0x7D00 в коде вызывается как 0xBD00.)


==============================================================================
 ЧАСТЬ 1. СЛОВАРЬ ЖИВЫХ ПЕРЕМЕННЫХ (ОЗУ)
 Это не «данные прошивки», а ячейки оперативки, куда ЭБУ кладёт текущие
 значения. Они — ключ: зная их, мы понимаем, ЧЕМ индексируется каждая таблица.
==============================================================================
   0x0F/0x10   = ОБОРОТЫ ДВИГАТЕЛЯ (RPM)             [ДОКАЗАНО: ищется по оси RPM карты]
   0x20/0x21   = НАГРУЗКА (Tp, базовый впрыск)        [ДОКАЗАНО: ищется по оси нагрузки]
   0x0D/0x0E   = РАСХОД ВОЗДУХА Q (сигнал ДМРВ)        [ДОКАЗАНО: вход VQ-таблицы]
   0x00FF      = ТЕМПЕРАТУРА ОЖ (охл. жидкости)        [ДОКАЗАНО: выход тарировки ТОЖ]
   0x007E,0x13 = ТЕМПЕРАТУРА ВОЗДУХА (второй сигнал ДМРВ) [ДОКАЗАНО: та же тарировка-термистор;
                 пороги декодируются в осмысленные °C]
   0x1500      = карта САМООБУЧАЮЩЕЙСЯ топл. коррекции в keep-alive ОЗУ [ДОКАЗАНО]
   0x8A        = масштабированный индекс для пачки 1D-кривых            [ГИПОТЕЗА: похоже температурный]

   Каналы АЦП (подпрограмма чтения $553A, всего 7):
     ch6 -> 0x00FF : датчик ОЖ                         [ДОКАЗАНО]
     ch3 -> 0x07   : опциональный датчик с диагностикой; кандидат — датчик темп. РОГ [ГИПОТЕЗА]
     ch4 -> 0x1058 : датчик, который ДИАГНОСТИРУЕТСЯ (см. часть 4)       [поведение ДОКАЗАНО, имя — гипотеза]
     ch5 -> 0x08   : в температурной группе; кандидат — IAT/темп. топлива [ГИПОТЕЗА]
     ch1 -> 0x0A   : аналог со сглаживанием; кандидат — ДПДЗ/МАФ          [ГИПОТЕЗА]
     ch2 -> 0x0B   : аналог со сглаживанием                              [ГИПОТЕЗА]
     ch7 -> 0x1057 : значение с обучением (мин/макс); кандидат — O2       [ГИПОТЕЗА]

   Общие подпрограммы (как ЭБУ читает таблицы):
     $80D2, $80DD = поиск по 1D-таблице (кривая/тарировка)
     $815D, $8021 = поиск по 2D-карте (RPM x нагрузка)
     $7096        = сравнение значения с таблицей-порогом (даёт «выше/ниже»)
     $553A        = чтение канала АЦП


==============================================================================
 ЧАСТЬ 2. ВАРИАНТНЫЕ (АЛЬТЕРНАТИВНЫЕ) КАЛИБРОВКИ
 Прошивка хранит НЕСКОЛЬКО версий некоторых таблиц и переключает их одним
 байтом-опцией 0x7EE0 (см. часть 5). В этом дампе байт = 0xFA, поэтому
 «альтернативы» сейчас выключены — оттого они и выглядели «мёртвыми».
==============================================================================
 0x79B0  тип: ТАБЛИЦА (64 точки, 16-бит)
         Что: АЛЬТЕРНАТИВНАЯ тарировка ДМРВ (вторая VQ-таблица).
         Включается, если бит 0x20 опции = 0. [ДОКАЗАНО]
         (Гипотеза зачем: под другой расходомер/рынок.)

 0x7AB0  тип: ОСЬ (16 точек)   Что: альтернативная ось НАГРУЗКИ для топл. карты. [ДОКАЗАНО]
 0x7AC0  тип: ОСЬ (16 точек)   Что: альтернативная ось ОБОРОТОВ для топл. карты. [ДОКАЗАНО]
         Включаются, если бит 0x10 опции = 0.


==============================================================================
 ЧАСТЬ 3. КАРТЫ И КРИВЫЕ КОРРЕКЦИИ
==============================================================================
 0x7BA0  тип: ОСЬ (16 точек)   \
 0x7BB0  тип: ОСЬ (16 точек)   /  Что: оси (RPM x нагрузка) для САМООБУЧАЮЩЕЙСЯ
         топливной коррекции. Сами данные коррекции — в ОЗУ 0x1500 (адаптируются
         на ходу). [ДОКАЗАНО, что это оси; что именно самообучение — сильно]

 0x7BF0  тип: 1D-КРИВАЯ (16 точек)
         Что: кривая коррекции. Вход — ОЗУ 0x1038, выход — ОЗУ 0x100E. [тип ДОКАЗАН]
         От чего именно зависит (что такое 0x1038) — [ГИПОТЕЗА], нужна доп. трассировка.

 0x7E00  тип: 1D-КРИВАЯ (16 точек)
         Что: УСЛОВНАЯ коррекция — работает только при включённом режиме
         (бит 1 флага 0x11). Вход — индекс 0x8A. [тип ДОКАЗАН; смысл — ГИПОТЕЗА]

 (рядом такие же безымянные 1D-кривые: 0x7E10, 0x7E30, 0x7E60, 0x7EA0 и др. —
  все читаются через $80D2 с индексом 0x8A; это коррекционные кривые, имена в
  XDF отсутствуют. [тип ДОКАЗАН, конкретный смысл — ГИПОТЕЗА])


==============================================================================
 ЧАСТЬ 4. ПОРОГИ И ДИАГНОСТИКА (большая категория!)
 Оказалось, почти все «мелкие/разрозненные» неизвестные куски — это БЛОКИ
 КОНСТАНТ-ПОРОГОВ, с которыми код сравнивает датчики и величины.
==============================================================================
 0x7780-0x77A7  тип: СКАЛЯРЫ-ПОРОГИ (~35 шт)
 0x77E0-0x77EC  тип: СКАЛЯРЫ-ПОРОГИ (~11 шт)
 0x7900-0x790A  тип: СКАЛЯРЫ-ПОРОГИ (~9 шт)
 0x7F10-0x7FE0  тип: СКАЛЯРЫ-ПОРОГИ + параметры (хвост; часть уже есть в XDF —
                отсечка, лимит скорости и т.д.)
   Что это всё: ПОРОГИ, которые ЭБУ использует в сравнениях. [тип ДОКАЗАН]
   Делятся на:
     - ПОРОГИ ДИАГНОСТИКИ: «если значение датчика вышло за предел и держится N
       циклов — поднять флаг неисправности». Пример доказан:
         * пара 0x7795=25 (низ) и 0x779B=235 (верх) для канала ch4 = проверка
           «обрыв/КЗ» датчика (значение АЦП < 25 или > 235 => ошибка).
         * 0x7798=3, 0x7797=2 = СЧЁТЧИКИ дебаунса (сколько циклов подтверждать).
     - ПОРОГИ ВКЛЮЧЕНИЯ ФУНКЦИЙ / режимов (по оборотам, нагрузке, температуре).
   Как читать значение порога: если он сравнивается с ОПОЗНАННОЙ переменной —
   применяем её формулу. Примеры (ДОКАЗАНО):
     * пороги по темп. воздуха (0x007E): 60->10°C, 65->15°C, 128->78°C,
       150->100°C, 155->105°C  (°C = байт - 50).
     * пороги по оборотам (0x0F) и по нагрузке (0x20) теперь тоже читаются
       физически (через известные оси).

 0x7E40 / 0x7F40  тип: ТАБЛИЦЫ-ПОРОГИ
   Что: значения, с которыми код сравнивает параметр через $7096 и ветвится
   (выше/ниже порога переключает логику). [тип ДОКАЗАН; что именно включают — ГИПОТЕЗА]


==============================================================================
 ЧАСТЬ 5. БЛОК КОНФИГУРАЦИИ / ОПЦИЙ
==============================================================================
 0x7ED9-0x7F2A  тип: ФЛАГИ/ОПЦИИ + пороги (раньше выглядел «шумом»)
   Что: НЕ таблица, а упакованные флаги настройки прошивки под вариант. Поэтому
   и читается побайтно, и выглядел хаотично. [ДОКАЗАНО]
   Ключевой байт 0x7EE0 = СЕЛЕКТОР ВАРИАНТОВ (в этом дампе = 0xFA):
     * бит 0x20 -> какую VQ (расходомер) использовать: основную или 0x79B0;
     * бит 0x10 -> какие оси топл. карты: основные или 0x7AB0/0x7AC0;
     * бит 0x08 -> включён ли канал ch3 (опциональный датчик).
   Остальные байты блока — прочие опции/пороги. [часть ДОКАЗАНА, часть ГИПОТЕЗА]


==============================================================================
 ЧАСТЬ 6. ОСТАЛОСЬ НЕЯСНЫМ (честно)
==============================================================================
 0x7200-0x7206 (7 байт)  : мелкие параметры, в коде глубоко не трассировали. [ГИПОТЕЗА]
 Точный физический смысл части 1D-кривых и нескольких каналов АЦП (ch1,ch2,ch5,ch7):
   поведение понятно (сглаживание/обучение/диагностика), но КАКОЙ это датчик —
   [ГИПОТЕЗА]. Чтобы закрыть — сверить каналы АЦП с распиновкой ЭБУ из мануала
   J30 (там список входов: ДПДЗ, темп. РОГ, O2, ГУР, скорость и т.д.).


==============================================================================
 ИТОГ ОДНИМ АБЗАЦЕМ
==============================================================================
 Почти все «неизвестные» данные — это НЕ загадки, а: (1) вариантные копии
 калибровок (вторая VQ, альт. оси), переключаемые байтом-опцией 0x7EE0;
 (2) оси самообучающейся коррекции; (3) безымянные кривые коррекции; и главное
 (4) БОЛЬШИЕ блоки скаляров-ПОРОГОВ для диагностики и включения функций.
 Структура (адрес/размер/тип) у всех известна — этого ДОСТАТОЧНО для своей XDF.
 Где не доказан физический смысл — стоит пометка [ГИПОТЕЗА].
==============================================================================



==============================================================================
 ЧАСТЬ 7. ЧТО МОЖНО ВНОСИТЬ В XDF ПРЯМО СЕЙЧАС
 Важно: для XDF нужно знать СТРУКТУРУ (адрес, размер, тип, формулу). Имя —
 желательно, но не обязательно. Ниже — два уровня готовности.
==============================================================================

 --- УРОВЕНЬ A: 100% (структура И смысл доказаны) — пишем с полным именем ---
   адрес    размер     тип     формула        имя в XDF
   0x79B0   64x1       16-бит  1.0*X          VQ Table (alternate)        [имя ДОКАЗАНО]
   0x7AB0   16x1       8-бит   X*0.125        TP Scale - Fuel (alt)       [имя ДОКАЗАНО]
   0x7AC0   16x1       8-бит   50*X           RPM Scale - Fuel (alt)      [имя ДОКАЗАНО]
   0x7EE0   1 байт     флаги   биты           Config/Options:
                                              b5=выбор VQ, b4=выбор осей,
                                              b3=вкл. ch3                 [мех-зм ДОКАЗАН]
   (формулы у альт-таблиц = такие же, как у их основных аналогов в готовом XDF.)

 --- УРОВЕНЬ B: структура 100% (XDF будет работать), имя — рабочее/гипотеза ---
   адрес    размер     тип     формула        рабочее имя
   0x7BA0   16x1       8-бит   как оси        Self-Learn Fuel Axis A      [ось ДОКАЗАНА]
   0x7BB0   16x1       8-бит   как оси        Self-Learn Fuel Axis B      [ось ДОКАЗАНА]
   0x7BF0   16x1       8-бит   X (сырое)      Correction Curve A          [тип ДОК., смысл гип.]
   0x7E00   16x1       8-бит   X              Correction Curve B (cond)   [тип ДОК., смысл гип.]
   0x7E40   16x1       8-бит   X              Threshold Table A           [тип ДОК.]
   0x7F40   16x1       8-бит   X              Threshold Table B           [тип ДОК.]
   блоки скаляров-порогов: 0x7780-0x77A7, 0x77E0-0x77EC, 0x7900-0x790A,
   часть 0x7F10-0x7FE0  — каждый байт = отдельный СКАЛЯР (8-бит).
   Часть из них уже с физикой:
     * пороги по темп. воздуха: формула (X-50) даёт °C        [ДОКАЗАНО]
     * пара 0x7795=25 / 0x779B=235 = диапазон «обрыв/КЗ» ch4   [ДОКАЗАНО]
     * 0x7798=3, 0x7797=2 = счётчики дебаунса диагностики      [ДОКАЗАНО]

 ВЫВОД: на УРОВНЕ A — можно сразу писать корректные именованные определения.
 На УРОВНЕ B — XDF тоже будет корректно ДЕКОДИРОВАТЬ (структура верна), просто
 имя пока рабочее; уточняется сверкой каналов АЦП с распиновкой ЭБУ из мануала.
