 РАЗБОР СЫРОГО ДАМПА ЭБУ — 88_Z31turbo32k.bin
 Nissan 300ZX (Z31) turbo, VG30ET, 1988.   Размер: 32768 байт = 32 КБ (27C256)
 Цель: показать, что можно отделить в сыром файле БЕЗ XDF, только анализом байт.

КАК ЧИТАТЬ ЭТОТ ФАЙЛ
  - "Факт"     = подтверждено формой/энтропией/картинкой (что это код/данные/ось/карта).
  - "Гипотеза" = смысл (какая карта — зажигание/топливо) и точные масштабы. Нужна
                 сверка с родственным XDF или дизассемблер.
  - Значения ниже даны в ДЕСЯТИЧНОМ виде (0..255), если не сказано иначе.
  - Адреса — это СМЕЩЕНИЕ В ФАЙЛЕ (то, что пишут в XDF).

 0. КАРТА ПОКРЫТИЯ ЗОНЫ ДАННЫХ 0x7200-0x7FFF  (полная, без пропусков)
------------------------------------------------------------------------------
  Проверено программно: регион разбит на сегменты, пропусков/наложений НЕТ,
  сумма байт = 3584 = 0xE00 (весь регион). Каждый адрес привязан к разделу.
  "->" указывает, в каком разделе ниже выписан этот участок.

    0x7200-0x7206  мелкие скаляры             -> F (прочее)
    0x7207-0x777F  заливка 0x3F               -> E
    0x7780-0x77A7  мелкие данные              -> F (прочее)
    0x77A8-0x77FF  заливка 0xFF               -> (заполнитель)
    0x7800-0x78FF  заливка 0x3F               -> E
    0x7900-0x790F  мелкие данные              -> F (прочее)
    0x7910-0x799F  заливки 0x3F / 0xFF        -> (заполнитель)
    0x79A0-0x79AF  заливка 0x3C (60)          -> E
    0x79B0-0x7AAF  ДМРВ/MAF: 2 кривые 16-бит   -> B5  [гипотеза]
    0x7AB0-0x7B2F  блок осей 8x16             -> B4
    0x7B30-0x7B4F  ТОЖ-тарировка (термистор)  -> B6  [гипотеза]
    0x7B50-0x7B5F  кривая-спад                -> B6
    0x7B60-0x7B6C  ось оборотов (0x7B61)      -> C
    0x7B6D-0x7B9F  заливки 232 / 0xFF / 0x00  -> (заполнитель)
    0x7BA0-0x7BBF  оси-индексы (2 шт)         -> C
    0x7BC0-0x7BCF  скаляр 64                  -> E
    0x7BD0-0x7BEF  заливка 0x3F               -> E
    0x7BF0-0x7BFF  кривая-спад                -> B6
    0x7C00-0x7CFF  КАРТА B1 (16x16)           -> B1
    0x7D00-0x7DFF  КАРТА B3+B2 (одна область) -> B2, B3
    0x7E00-0x7E6F  кривые                     -> D
    0x7E70-0x7E7F  заливка 0x00               -> (заполнитель)
    0x7E80-0x7E8F  кривая                     -> D
    0x7E90-0x7E9F  ось/кривая 0x7E90          -> C
    0x7EA0-0x7EDF  кривые                     -> D
    0x7EE0-0x7F2F  шум                        -> F
    0x7F30-0x7F6F  кривые                     -> D
    0x7F70-0x7FDF  смесь / шум                -> F (прочее)
    0x7FE0-0x7FFF  хвост / векторы            -> G

  ВАЖНО: полные байты ВСЕХ участков лежат в разделе H (непрерывный дамп).
  Разделы A-G — это подписи поверх H. Если где-то подписи нет — байты всё
  равно есть в H, потерять данные нельзя.

 A. ЗОНЫ ФАЙЛА (крупное деление)                                      [ФАКТ]
------------------------------------------------------------------------------
  0x0000 - 0x3FFF   ПУСТО. Сплошь 0xFF (16384 б). ПЗУ мапится в верх адресации.
  0x4000 - 0x71FF   КОД программы. Шум, энтропия ~6.0, ср.|дельта|~90. ~12.5 КБ.
  0x7200 - 0x7FFF   КАЛИБРОВКА (данные): оси, карты, кривые, скаляры, векторы.

  Как выглядит граница "код -> данные":
    КОД   (0x04210): E5 22 7F 91 C3 04 B7 08 EA 5F 12 9D 33 7C A1 6B   <- хаос
    ДАННЫЕ(0x07C40): 25 25 32 32 31 31 31 31 31 31 25 20 15 13 09 09   <- плавно

  Внутри данных большая заливка-дефолт:
    0x7207 .. ~0x7787  = 1401 байт значения 0x3F (63). "Пустые" ячейки калибровки.

        @0x79B0 (64 точки, до 0x7A2F): 110 147 193 248 313 390 478 579 694 ... 62501 65535

 E. СКАЛЯРЫ / КОНСТАНТНЫЕ БЛОКИ                                       [ФАКТ]
------------------------------------------------------------------------------
  Ряды одинаковых значений = пороги/дефолты/заполнители:
    0x3F (63)  — гигантская заливка 0x7207.. + блоки. Самый частый байт (после FF).

 G. ВЕКТОРЫ / ПРОЦЕССОР (хвост файла)                          [ФАКТ + вывод]
------------------------------------------------------------------------------
  0x7FF8 - 0x7FFE — кандидаты в векторы прерываний/reset:
        0x7FF8: 91 C7     0x7FFA: 83 CE     0x7FFC: B2 00     0x7FFE: 83 CE
  При чтении LITTLE-ENDIAN они указывают В ЗОНУ КОДА:
        0xC791 -> файл 0x4791 (код)
        0xCE83 -> файл 0x4E83 (код)
  ВЫВОД: ПЗУ мапится в верхнюю половину 64К-адресации
         (CPU-адрес = смещение_в_файле + 0x8000).
  ПОПРАВКА по endianness: байтовые 16-битные ТАБЛИЦЫ (раздел B5, @0x79B0) явно
  big-endian (старший байт первый, ряд монотонно растёт). А "векторы" в хвосте
  красиво разрешаются только как little-endian. Это противоречие НЕ разрешено
  без дизассемблера: либо хвост — не векторы, либо карта памяти иная.
  Честный статус: endianness данных = big-endian (ФАКТ по B5);
  трактовка хвоста как векторов = гипотеза.

  Из "набора цифорок" без подсказок отделено:
    - 3 зоны: пусто (16КБ) / код (~12.5КБ) / калибровка (~3.5КБ);
    - ~4 явные 2D-карты + блок осей;
    - 3 блока 16-битных таблиц (0x79B0-0x7AAF, big-endian);
    - 3-4 оси (1D);
    - ~14 кривых-коррекций (1D);
    - россыпь скаляров/констант (дефолт 0x3F);
    - 1 неразобранный шумовой блок (указатели/код);
    - векторы в хвосте + тип адресации/endianness процессора.

  ФАКТ  : где код, где данные, где таблицы/оси/кривые/константы/векторы.
  ГИПОТЕЗА: какая карта = зажигание, какая = топливо; точные scale/offset.
  Добивается: (1) наложить родственный XDF -> мгновенная подпись, либо
              (2) дизассемблировать код -> по каким адресам он читает таблицы.

  Метод (чем получено): hex-анализ + энтропия блоков + поиск возрастающих рядов
  (оси) + поиск гладких многорядных блоков (карты) + рендер "байт=пиксель"
  (картинка /tmp/z31_map.png: белый верх = пусто, "снег" = код, полосы внизу = карты).
