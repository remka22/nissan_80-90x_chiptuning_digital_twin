# Реверс прошивок ЭБУ Nissan ECCS — полная методичка (playbook)

Этот файл — пошаговый разбор всего, что было проделано: от голых байтов прошивки
до дизассемблирования кода и распознавания неизвестных данных. По нему можно
повторить работу с нуля (в т.ч. обучить новую сессию ассистента), не изобретая
заново.

**Объект:** ЭБУ Nissan ECCS на процессоре **Hitachi HD6303** (8-бит, совместим с
Motorola 6801/6301, **big-endian**), внешняя EPROM 32 КБ (27C256).
**Примеры:** `z31/88_Z31turbo32k.bin` (300ZX Z31 VG30ET 88), `m30/infiniti90m30at (1).bin`
(Infiniti M30 90-92 VG30E).

**Инструмент:** `tools/dis6303.py` — дизассемблер 6303 + поиск ссылок/писателей.

---

## Главная идея (в одну строку)
Прошивка = **КОД** (программа процессора) + **ДАННЫЕ** (калибровка: карты, оси,
кривые, скаляры). Сначала вытаскиваем структуру данных по форме, затем (если есть)
расшифровываем готовым XDF, а для непонятного — читаем КОД, который эти данные
использует. Смысл данных = «чем их индексируют (вход) и куда идёт результат (выход)».

## ФАЗА 1. Разбор сырых данных (без XDF)

Цель: отделить код от данных и опознать таблицы ПО ФОРМЕ.

1. **Размер и гистограмма байтов.** 32768 = 32КБ. Самый частый байт после `0xFF` —
   обычно `0x3F` (заполнитель пустых ячеек калибровки).

2. **Карта зон по энтропии** (блоки по 256–512 б):
   - `H≈6`, «телевизионный снег» → **КОД**.
   - один и тот же байт (`0xFF`/`0x3F`/`0x00`) → **ЗАЛИВКА** (пусто).
   - низкая энтропия + плавные дельты → **ДАННЫЕ** (таблицы).
   У Z31: пусто 0x0000-0x3FFF, код 0x4000-0x71FF, данные 0x7200-0x7FFF.
   У M30: код с 0x0000, заливка в середине, данные 0x6A00 и 0x7400-0x7FFF.

3. **Сигнатуры таблиц (запомнить!):**
   - **2D-карта** — несколько подряд идущих ПОХОЖИХ рядов с плавными значениями.
   - **карта топлива (AFR)** — «рич-квадрат» из байтов `192-199` (0xC0) в углу:
     старший бит = флаг «замкнутый цикл». Формула AFR: `1881.6/(X+128)` (откр.цикл)
     или `1881.6/(X-64)` (замкн.). Константа `1881.6 = 14.7 * 128` (привязка к стехиометрии).
   - **карта зажигания** — байты = градусы напрямую (диапазон ~1..50), формула `1.0*X`.
   - **VQ / ДМРВ (расходомер)** — **16-битная** монотонная ЭКСПОНЕНТА до `0xFFFF`
     (big-endian пары байт). 64 точки.
   - **тарировка термистора (темп. ОЖ/воздуха)** — гладкий монотонный СПАД (255→0),
     ~32 точки; формула обычно `X-50` (°C).
   - **ось / брейкпоинты** — короткий ВОЗРАСТАЮЩИЙ ряд (обороты/нагрузка).
   - **1D-кривая коррекции** — один ряд, монотонный (спад/рост).
   - **скаляр/порог** — одиночные константы; блоки одинаковых значений.

4. **Трюк «байт = пиксель».** Отрендерить дамп как ч/б картинку (ширина 64):
   карты видны гладкими прямоугольниками, код — шумом, заливка — однотонным полем.

5. **Полный дамп зоны данных** (важно: НЕ сокращать, выписывать всё; длинные
   заливки можно сворачивать пометкой «повтор xN», это не потеря данных).

6. **Карта покрытия** — разбить зону данных на сегменты так, чтобы КАЖДЫЙ байт попал
   ровно в один (проверка: сумма = размер зоны, нет дыр). Тогда нельзя «потерять» таблицу.

Результаты Z31 в `z31/Z31_dump_разбор.txt`, M30 в `m30/M30_dump_разбор.txt`.

## ФАЗА 2. Расшифровка по XDF

XDF — файл определений для TunerPro: говорит, по какому адресу что лежит и как
пересчитать байт в физику. Здесь — **legacy-формат 1.10** (блоки `%%TABLE%%`).

1. **Парсинг XDF.** Для каждого `%%TABLE%%`/`%%CONSTANT%%` берём: `Title`, `Address`,
   `SizeInBits`, `Rows`, `Cols`, `Equation/ZEq`, `ZUnits`. (Парсер — простой regex.)
   `BinSize` и `BaseOffset=0` → **адрес в XDF = смещение в файле напрямую**.

2. **Размер ячейки** = `SizeInBits/8` (нет поля → 8 бит). Спан = `Rows*Cols*bytes`.

3. **Применить формулу** к каждой ячейке → физические значения. Осторожно: `ZUnits`
   (единица ЯЧЕЙКИ) ≠ `XUnits` (единица ОСИ). Пример: у топл.карты ось `TP (ms)` —
   это нагрузка, а ЯЧЕЙКА — AFR.

4. **Проверка.** Сверить с ожидаемым (ХХ ~750 об/мин, УОЗ ~15°, отсечка, и т.д.).
   Если значения «дикие» (5.75 мс там, где должно быть ~14) — применена НЕ та формула
   к части ячеек (см. флаг 0xC0 у топл.карты: каждую ячейку считать своей формулой).

Результаты: `*_расшифровка_по_XDF.txt`, `*_таблицы_decoded.txt`.

## ФАЗА 3. Сравнение двух прошивок

Один двигатель ≠ один ЭБУ. Но в семействе ECCS разметка часто почти совпадает.

- Сопоставить адреса таблиц: что ОДИНАКОВО, что НОВОЕ, что УШЛО, что СМЕСТИЛОСЬ.
- Пример: M30 = Z31 + **детонационная пара** карт (Knock Timing `0x7800` + Knock Fuel
  `0x7400`), потому что у M30 появился ДАТЧИК ДЕТОНАЦИИ. Подтверждается и числами
  (Knock-зажигание имеет МЕНЬШЕ угла, Knock-топливо БОГАЧЕ в зоне детонации), и
  именами в XDF («Knock …»).
- Полезно для переноса определений между прошивками.

Результат: `Переход_Z31_к_M30.txt`.

## ФАЗА 4. Дизассемблирование (когда XDF не хватает)

Нужно, чтобы понять НЕОПИСАННЫЕ данные: кто их читает и зачем.

### 4.1 Определить процессор и карту памяти
- Процессор — по **маркировке главного чипа** на плате: `HD63B03Y` = HD6303 (наш случай).
- **BASE (смещение ROM в адресах CPU):** запустить `find_base()` — гистограмма
  адресов в инструкциях `LDX #imm` (опкод `CE`). Где густо — там таблицы.
  Данные в файле в `0x7xxx`; если в коде они адресуются как `0xBxxx` → **BASE=0x4000**
  (Z31); если `0xFxxx` → **BASE=0x8000** (M30).
  Формула: **CPU-адрес = смещение_в_файле + BASE**.

### 4.2 Опорные опкоды 6303 (хватает для чтения логики)
```
CE = LDX #imm16     DF = STX (zero-page)    FE/DE = LDX ext/dir
96/D6 = LDAA/LDAB (zero-page)   B6/F6 = LDAA/LDAB ext
97/D7 = STAA/STAB (zero-page)   B7/F7 = STAA/STAB ext
B1/F1 = CMPA/CMPB ext           81 = CMPA #imm     85 = BITA #imm
BD = JSR ext     8D = BSR     7E = JMP     39 = RTS    7D = TST ext
20..2F = ветвления (BRA/BNE/BEQ/BCC/BCS/BHI/BLS/...)
```

### 4.3 ГЛАВНЫЙ ПРИЁМ — «якоря по адресам»
В коде обращение к таблице — это `LDX #её_CPU_адрес`. Значит:
- `refs(0x7D00)` → найдёт, ГДЕ код грузит топл.карту (file 0x7D00 → cpu 0xBD00).
- Если рядом грузятся ДВЕ таблицы (например `0xBD00` и `0xB400`) — это **выбор из
  двух** (нормальная/детонационная, основная/альтернативная). Дизассемблировать это
  место — увидишь условие выбора.

### 4.4 Опознание данных через код
- **Выбор варианта:** `LDAA <конфиг-байт> ; BITA #маска ; BNE/BEQ ; LDX #таблицаA / #таблицаB`.
  Конфиг-байт (у Z31 — `0xBEE0` = file `0x7EE0`) своими битами переключает калибровки.
- **Чтение датчика (АЦП):** паттерн `LDAA #номер_канала ; BSR <ADC-routine>` ; затем
  `STAA <RAM>`. Так строится список «канал → RAM-переменная → датчик».
- **Тарировка:** значение датчика идёт `LDX #таблица_тарировки ; JSR <1D-поиск>` ;
  результат в RAM. (У нас: ОЖ → 0x00FF; вторая температура (воздух из ДМРВ) → 0x007E.)
- **Карта RPM×нагрузка:** в подпрограмме 2D-поиска (`$815D`) видно, какие RAM —
  индексы: один ищется по оси оборотов, другой по оси нагрузки.
  У Z31: **0x0F/0x10 = обороты**, **0x20/0x21 = нагрузка (Tp)**, **0x0D/0x0E = расход Q**.
- **Диагностика/пороги:** `LDAA <датчик> ; CMPA <ROM-порог> ; ветвление` ; счётчик
  дебаунса; при стойком выходе за предел — `ORAA #бит ; STAA <флаг>`. Пара
  «низкий/высокий порог» на один датчик = проверка «обрыв/КЗ».

### 4.5 Строим СЛОВАРЬ RAM-переменных
Опознав датчики (по тарировкам-якорям) и индексы карт (RPM/нагрузка), получаем
словарь: какая RAM-ячейка что содержит. Дальше смысл ЛЮБОЙ таблицы «выпадает» из
того, чем её индексируют и куда кладут результат.

Итог распознавания Z31: `z31/Z31_код_распознанное.txt`.

## Шпаргалка-сигнатуры (быстрый опознаватель)

| Что видишь в данных | Это, скорее всего |
|---|---|
| 16-бит экспонента до 0xFFFF, 64 точки | VQ / тарировка ДМРВ |
| гладкий спад 255→0, ~32 точки | тарировка термистора (темп. ОЖ/воздуха), °C = X-50 |
| «квадрат» из байтов 192-199 (0xC0) | карта топлива (AFR); старший бит = замкнутый цикл |
| гладкая 16×16, значения ~1..50 | карта зажигания (градусы, 1.0*X) |
| короткий возрастающий ряд (8,16,24…) | ось / брейкпоинты (обороты или нагрузка) |
| один монотонный ряд | 1D-кривая коррекции |
| блок одинаковых байтов / россыпь констант | скаляры/пороги |
| упакованные «шумные» байты, читаются LDAA (не LDX#) | конфиг/опции или указатели |
| заливка 0x3F / 0xFF / 0x00 | пусто |

## Дисциплина «доказано vs гипотеза»
- **Доказано** — подтверждено кодом/формой/проверкой. Помечать `[ДОКАЗАНО]`.
- **Гипотеза** — по косвенным признакам. Помечать `[ГИПОТЕЗА]` и писать, какого шага
  не хватает (обычно — опознать RAM-переменную или сверить АЦП-канал с распиновкой ЭБУ).
- Не выдавать догадку за факт. Если передумал — объяснить, что и почему было неверно.
- Когда пользователь просит «выписать данные» — выписывать ПОЛНОСТЬЮ, без сокращений.
