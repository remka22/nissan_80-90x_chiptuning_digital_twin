# Инструкция: дизассемблинг и декомпиляция прошивок ECCS (HD6303)

Процессор всех трёх блоков — **Hitachi HD6303** (big-endian, 6801/6301-совместимый).
Два уровня инструментов: быстрый самописный дизассемблер и полноценная Ghidra с C-декомпилятором.

---

## 0. Карта памяти (КРИТИЧНО — определяет базу загрузки)

ROM 32 КБ. CPU-адрес = смещение_в_файле + BASE. BASE определяется по reset-вектору:
читаем последние 2 байта файла (0x7FFE/0x7FFF) — это CPU-адрес старта; при правильной
базе по этому адресу должен быть осмысленный код (обычно `LDX #..`, `SEI`, очистка RAM).

| Прошивка | Файл | reset-вектор | **BASE** | Код (CPU) | Вектора (CPU) |
|---|---|---|---|---|---|
| M30  | infiniti90m30at (1).bin | 0xB06F | **0x8000** | 0x8000–0xC9FF | 0xFFF0–0xFFFF |
| R32  | R32_GTS4_stock_ROM...bin | 0xB1D8 | **0x8000** | 0x8000–0xBFFF | 0xFFF0–0xFFFF |
| Z31  | 88_Z31turbo32k.bin | 0x83CE | **0x4000** | 0x8000–0xB3FF | **0xBFF0–0xBFFF** |

ВНИМАНИЕ: у **Z31 база 0x4000** (код в верхней половине файла), и таблица векторов
лежит на CPU **0xBFF0–0xBFFF** (вершина блока), а не на 0xFFF0. У M30/R32 — стандартно 0x8000.
Как проверить: дизассемблировать reset-цель при обеих базах — где код, а не FF-заливка.

## 1. Быстрый путь — самописный `tools/dis6303.py`

Подходит для точечного анализа (одна процедура, поиск ссылок), не требует установки.

```bash
# дизассемблировать N инструкций с file-offset (НЕ cpu!)
python3 tools/dis6303.py "m30/infiniti90m30at (1).bin" 0x8000 dis <file_off> <N>
# найти ссылки на ROM-таблицу
python3 tools/dis6303.py <bin> <base> refs <cpu_addr>
# кто пишет переменную
python3 tools/dis6303.py <bin> <base> writers <cpu_addr>
```
Помни про 6301-расширения, которые базовый декодер может не знать:
`0x18=XGDX`, `0x3D=MUL`, биты `0x71 AIM / 0x72 OIM / 0x75 EIM / 0x7B TIM #imm,$dp`
(set/clear/test бита — ими управляются флаги и соленоиды), индексные формы `0x61/62/65/6B`.

## 2. Полный путь — Ghidra + C-декомпилятор (как делали)

### 2.1 Установка (один раз)
```bash
brew install openjdk@21
# Ghidra 12.0.4 (под версию модуля):
curl -sL -o ghidra.zip "https://github.com/NationalSecurityAgency/ghidra/releases/download/Ghidra_12.0.4_build/ghidra_12.0.4_PUBLIC_20260303.zip"
unzip -q ghidra.zip
# Модуль HD6301/HD6303 (community, sigurasg/GhidraMC6800), сборка под 12.0.4:
curl -sL -o module.zip "https://github.com/sigurasg/GhidraMC6800/releases/download/v1.0/ghidra_12.0.4_PUBLIC_20260304_GhidraMC6800.zip"
unzip -o -q module.zip -d ghidra_12.0.4_PUBLIC/Ghidra/Extensions/
```
Язык процессора: **`HD6301:BE:16:default`** (BE = big-endian; покрывает HD6303).
Всё установлено в `~/ghidra-eccs/`. Скрипты — в `~/ghidra-eccs/scripts/`.

### 2.2 Скрипты (Java — в Ghidra 12 Jython убран, нужен Java или PyGhidra)
- `PrepEntries.java` — preScript: читает вектора (M30/R32 на 0xFFF0, Z31 — `PrepEntriesZ31.java` на 0xBFF0),
  создаёт функции на их целях, разбирает код от старта.
- `DecompAll.java` — postScript: декомпилирует все функции в C → файл из env `DECOMP_OUT`.
- `ListingAndEgr.java` — выгружает полный дизасм-листинг + адресно создаёт функции.

### 2.3 Запуск (headless)
```bash
# M30 / R32 (base 0x8000):
JAVA_HOME=/opt/homebrew/opt/openjdk@21 DECOMP_OUT=/tmp/out.c \
~/ghidra-eccs/ghidra_12.0.4_PUBLIC/support/analyzeHeadless ~/ghidra-eccs/proj <PROJ> \
  -import "<путь к bin>" -processor "HD6301:BE:16:default" \
  -loader BinaryLoader -loader-baseAddr 0x8000 \
  -scriptPath ~/ghidra-eccs/scripts -preScript PrepEntries.java -postScript DecompAll.java

# Z31 — base 0x4000 + PrepEntriesZ31.java
```
Папка проекта (`~/ghidra-eccs/proj`) должна существовать заранее (`mkdir -p`).

### 2.4 Результат (уже в проекте)
| Прошивка | Файл декомпиляции | Функций |
|---|---|---|
| M30 | `m30/M30_декомпиляция_C.c` (+ листинг `.asm`, EGR-актуаторы) | 105 |
| R32 | `r32/R32_декомпиляция_C.c` | 106 |
| Z31 | `z31/Z31_декомпиляция_C.c` | 53 |

## 3. Как читать вывод
- Переменные `DAT_xxxx` = адреса RAM/ROM. Наши имена: `DAT_1460`=обороты, `DAT_1424`=ОЖ,
  `DAT_1413`=нагрузка, `DAT_00ab/00a9/00a7`=байты-соленоиды, таблицы `0x7B30`=термистор ОЖ и т.д.
- `FUN_xxxx` = функция по CPU-адресу. `in_D`/`CONCAT`/`undefined` — артефакты декомпилятора, норма для 8-бит.
- Узнаваемые: интерполяция таблиц, термистор-lookup, MUL-математика, оконные соленоиды (EGR).

## 4. GUI-навигация (опционально)
Проект уже создан в `~/ghidra-eccs/proj`. Открыть `ghidraRun`, импортированные программы
(M30j/R32j/Z31b) — кликабельные xref, переименование, граф вызовов.

## 5. Известные нюансы
- Ghidra 12 без PyGhidra не исполняет `.py`-скрипты — только `.java` (или включать PyGhidra).
- Редкие pcode-ошибки «Unable to resolve constructor» = модуль не знает отдельный опкод/наткнулся
  на данные — на общую декомпиляцию почти не влияет.
- Z31 (старое поколение) даёт меньше функций и почти не использует прямые AIM/OIM/TIM
  (см. сравнение «породы» кода — TIM 5 раз против 400–488 у M30/R32).
