# Инструкция: патчинг прошивок ECCS (HD6303) и контрольная сумма

Цель — **изменить прошивку и получить рабочий ROM**. Перекомпиляция декомпилированного
C обратно в образ НЕ делается (адреса RAM/портов/векторов жёстко зашиты, образ не заведётся).
Патчат **существующий бинарь на месте** — двумя способами + обязательный пересчёт контрольной суммы.

---

## ПУТЬ А — правка калибровок (ДАННЫЕ). Основной, безопасный.

Это смысл XDF: меняешь байты таблиц/констант в зоне данных. Логика кода не трогается.

1. Найди адрес параметра (= смещение в файле) по нашим файлам:
   - карты/оси/датчики: `*_таблицы_decoded.txt`, `*_расшифровка_по_XDF.txt`;
   - соленоиды/окна EGR: `m30/M30_соленоиды_EGR_для_XDF.txt`;
   - скаляры (отсечка, пороги O2, Base Idle и т.д.): инвентари `*_XDF_инвентарь.txt`.
2. Поменяй байт(ы) по формуле параметра (например RPM=50·X → отсечка 7000 = X=140=0x8C).
3. **Пересчитай контрольную сумму** (раздел ниже).
4. Прошей.

Пример типовых правок: отсечка (скаляр), карта УОЗ/топлива (16×16), кромки окна EGR
(`0x779E/0x7695` и т.д.), Base Idle RPM (`0x76B0`).

---

## ПУТЬ Б — правка ЛОГИКИ (КОД). Хирургический asm-патч.

Если надо поменять не число, а поведение.

1. **Свободное место** есть: зона `0x4000–0x73FF` (file) — сплошной `0xFF` (~14 КБ пусто)
   у M30 и R32. CPU-адрес: M30/R32 = file+0x8000; у Z31 базы 0x4000 — смотри карту памяти.
2. Пишешь свой код HD6303-опкодами в пустую зону.
3. В нужном месте старого кода врезаешь переход на вставку:
   - `JMP $addr` (опкод `7E hi lo`) или `JSR $addr` (`BD hi lo`);
   - на 3 байта оригинала — ровно влезает `JMP`/`JSR`; вытесненные инструкции
     переноси в начало вставки, затем возвращайся (`JMP back`).
4. Проверь вставку дизассемблером (`tools/dis6303.py ... dis`).
5. **Пересчитай контрольную сумму**. Прошей.

Опоры по опкодам: `LDAA #imm=86`, `LDAA $ext=B6`, `STAA $ext=B7`, `CMPA #=81`,
`JMP=7E`, `JSR=BD`, `RTS=39`, `BNE=26`, `BRA=20`, биты `OIM #b,$dp=72 / AIM=71 / TIM=7B`.

## КОНТРОЛЬНАЯ СУММА (обязательно после любой правки)

### M30 и Z31 — алгоритм НАЙДЕН и ПОДТВЕРЖДЁН
Процедура (M30 @ cpuB720): бежит по **всему ROM** и считает две суммы,
сверяет с эталонами; не сойдётся — уход в error-ветку.

- **XOR** всех байтов ROM  → эталон в байте `0xFF7A` (file **0x7F7A**)
- **SUM (8-бит)** всех байтов ROM → эталон в байте `0xFF7B` (file **0x7F7B**)

Проверка на стоке:
- M30: XOR=0x2E, SUM=0x32 — совпало с байтами 0x7F7A/0x7F7B. ✓
- Z31: XOR=0xDD, SUM=0xF5 — совпало. ✓ (та же схема, у Z31 это CPU 0xBF7A/0xBF7B)

### Формула починки после патча (выведена и проверена)
Оба контрольных байта сами входят в суммы, поэтому чинится в одно действие.
Считаешь по всему ROM **кроме** позиций 0x7F7A и 0x7F7B:
```
XO = XOR всех байтов, кроме 0x7F7A и 0x7F7B
SO = SUM всех байтов (mod 256), кроме 0x7F7A и 0x7F7B
byte[0x7F7A] = (-SO) & 0xFF      # чинит SUM-условие
byte[0x7F7B] = XO                # чинит XOR-условие
```
Проверено: на стоке M30 `XO=0x32, SO=0xD2` → `byte[7F7A]=0x2E`, `byte[7F7B]=0x32` (= сток).

### Готовый скрипт `fix_checksum.py` (для M30 и Z31)
```python
import sys
p = sys.argv[1]
d = bytearray(open(p,'rb').read())          # 32 КБ ROM
A, B = 0x7F7A, 0x7F7B                        # позиции эталонов
xo = so = 0
for i,b in enumerate(d):
    if i in (A,B): continue
    xo ^= b; so = (so + b) & 0xFF
d[A] = (-so) & 0xFF                          # SUM-эталон
d[B] = xo                                    # XOR-эталон
open(p,'wb').write(d)
print("checksum fixed: 0x7F7A=%02X 0x7F7B=%02X" % (d[A], d[B]))
```
Запуск: `python3 fix_checksum.py путь_к_патченному.bin`

### R32 — схема ДРУГАЯ (внимание!)
У R32 XOR/SUM по всему ROM НЕ ложатся на 0x7F7A/0x7F7B (там C4/EC, а суммы FF/B1).
Перед патчем R32 нужно **отдельно найти его checksum-процедуру** тем же методом:
искать цикл-сканер ROM (`INX → EORA/ADDB 0,X → CPX #$FFFF → BNE`) и сравнения
`CMPA $store / CMPB $store`. Диапазон может исключать пустую зону 0x4000–0x73FF.
Пока не подтверждён — R32 не патчить.

## Порядок безопасной работы
1. Сделать резервную копию стокового бинаря.
2. Внести правку (данные или код).
3. Пересчитать контрольную сумму (`fix_checksum.py` для M30/Z31).
4. Сверить дизассемблером, что патч-код корректен (для пути Б).
5. Прошить; если ЭБУ уходит в ошибку/не заводится — почти всегда не пересчитана сумма
   или задет жёстко-зашитый адрес.
